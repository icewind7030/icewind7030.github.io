去年年底针对蜗牛的前端开发业务特点，搭建了一个静态资源工程，一方面是为了将蜗牛的日常前端开发工作尽量和后端的耦合分离，加快开发部署的效率；另一方面也是为了增加一些工程化方面的特性，比如分支规范、代码检查等。

工程介绍

功能

本静态资源工程功能很简单，就是维护了蜗牛前端开发中的纯静态资源，包含将要部署的页面和页面所依赖的资源。目的就是将蜗牛之前与后段工程耦合在一起的前端开发工程独立出来。

原理

静态资源本地开发构建完成后，通过NDP平台将要部署的文件打包发布到NOS上后，再由集群上的主机去NOS上拉取资源包然后解压并存储，最后再通过服务器上的Nginx转发到本机的存放目录。类似下面的Nginx配置：

就使得所有蜗牛域名的/fe路径下URL请求都转发到了这台机器上的/home/frontend-static/目录下的对应文件位置。

结构

本工程结构主要针对蜗牛读书前端开发业务较为离散、独立的特点而设计，目录结构主要有3类:

  • 工具类——存放构建和工程相关文件
  • 共用类——存放前端开发中常用和共用的资源，部署时将被原样部署
  • 业务类——存放活动、Hybrid和较为独立的功能页面等，以文件夹为管理单位，通过package.json中的publishFolders配置文件夹内部所需部署的目录

构建

本工程采用的是本地构建的构建方式，也就是各个业务文件夹自行配置和使用webpack或者其他构建工具在本地直接进行开发和构建。这么设计的主要原因是本工程的用途比较多样，技术栈暂时也没有统一的必要，本地部署的灵活性和稳定性更高。

部署

部署逻辑主要有以下几个特点：

  • 避开了NDP较为复杂的Ant脚本开发，而是通过Ant调用NodeJS脚本的方式工作
  • 工程中附带了较高版本的Node环境，避免构建机中Node环境特性不足以及升级构建机中Node对其他工程的影响
  • 结合Git命令，获取本次部署和上一次的差异，从而实现最小化的部署开销，加快部署速度
  • 识别工程内页面下package.json中的配置字段，只部署打包后的文件，避免暴露源码，减少部署开销和时间

NDP平台采用的实例模板为static_append，即对已存在的同名资源覆盖、非同名资源增加的方式，不需要重启服务，所以效率和稳定性都比较高。

得益于本地构建、资源覆盖和差量部署几个特点，蜗牛读书的前端静态资源工程的一次部署只需要几秒钟。

差量部署原理：在NDP构建机上，利用构建脚本，查询一个在.gitignore列表中的配置文件（意味着该文件不会被上传和覆盖，只存在与构建机）。如果文件存在，则读取该文件中保存的上一次部署时工程的Git Hash，再结合本次部署的Git Hash，利用git diff命令，获得到两次hash间的文件差异从而进行最小化的差量部署；如果文件不存在或者两次hash相同，则进行全量部署，并且生成或更新配置文件。

附加功能

前端工程分离出来以后，就更便于添加一些其他的功能，比如分支规范检查、代码规范检查等

Git Hooks

Git Hooks是Git作为一个版本控制系统提供的在某些特定时机可以被触发的钩子，我可以在这些时机执行一些自定义脚本，这些自定义脚本可以做一些校验的操作，如果脚本以非零值退出，那么钩子将阻止本次操作。
Git Hooks有两类：客户端的和服务端的，客户端的钩子并不随代码提交，只在本地运行，而服务端的钩子只在远端工作。

常用客户端钩子

  • pre-commit 钩子在键入提交信息前运行。
  • prepare-commit-msg 钩子在启动提交信息编辑器之前，默认信息被创建之后运行。
  • commit-msg 在用户提交信息后触发，并且可以读取到提交的信息。
  • post-commit 钩子在整个提交过程完成后运行。

    常用服务端钩子

  • pre-receive 服务端收到来自客户端的推送操作时触发。
  • update 和 pre-receive 十分类似，不同之处在于如果一次推送到多个分支，update钩子会在每个分支都执行一次，而 pre-receive 只执行一次。
  • post-receive 在整个推送完结以后运行，可以用来更新其他系统服务或者通知用户。

提交信息的检查

有了Git Hooks之后，我们就可以对开发人员提交的Git Commit进行格式的检查，拒绝不符合格式的提交信息。
这里我们使用的就是commit-msg钩子。

Husky连接的钩子可以接收2个参数：

  • HUSKY_GIT_PARAMS代表该Git钩子运行时传入的参数，比如这里commit-msg钩子中就会有本次提交的message文件存放位置；
  • HUSKY_GIT_STDIN代表该Git钩子运行时从标准输入stdin中可以读取的一些信息，比如pre-push钩子中就会有一系列待更新的Git引用(git ref)

拿到commit message文件地址后，我们就可以读取本次提交的message内容了，配合以下我们团队制定的commit message规则进行校验即可：

  • feat：新功能（feature）
  • fix：修补bug
  • docs：文档（documentation）
  • style： 格式（不影响代码运行的变动）
  • refactor：重构（即不是新增功能，也不是修改bug的代码变动）
  • test：增加测试
  • chore：构建过程或辅助工具的变动

提交代码的检查

ESLint

我们使用ESLint可以配合Git Hooks进行代码检查，这里简要介绍一下Eslint：

重要概念
  • env 脚本的运行环境。每种环境都有一组特定的预定义全局变量,比如browser环境下的window
  • globals 脚本在执行期间访问的额外的全局变量，比如jquery的$
  • rules 启用的规则及其各自的错误级别，比如"no-console": 0/1/2/"off"/"warn"/"error"
  • plugins ESLint的插件，它可以是以上3类配置的任意组合，通过插件内的configs字段进行定义，而且configs字段还可以包含多个配置级别，比如all，代表各级别通用，recommended代表推荐级别，比all要多出一些规则和配置

rules在定义时，是通过分析Javascript代码转换成的AST树来实现的，这使得规则不但可以判断错误，还可以定义如何修复错误，一旦我们使用的ESLint命令行包含了–fix选项，ESLint就可以根据规则内置的错误修复逻辑直接帮我们修复错误。

配置文件

ESLint 支持几种格式的配置文件：

  • JavaScript - 使用 .eslintrc.js 然后输出一个配置对象。
  • YAML - 使用 .eslintrc.yaml 或 .eslintrc.yml 去定义配置的结构。
  • JSON - 使用 .eslintrc.json 去定义配置的结构，ESLint 的 JSON 文件允许 JavaScript 风格的注释。
  • (弃用) - 使用 .eslintrc，可以使 JSON 也可以是 YAML。
  • package.json - 在 package.json 里创建一个 eslintConfig属性，在那里定义你的配置。
    如果同一个目录下有多个配置文件，ESLint 只会使用一个。优先级顺序如下：
  1. .eslintrc.js
  2. .eslintrc.yaml
  3. .eslintrc.yml
  4. .eslintrc.json
  5. .eslintrc
  6. package.json

使用插件

ESLint支持的检查规则有非常的多，一个个去配置显然并不现实，所以我们需要使用插件。
ESLint的插件通常都是NPM包，我们需要先安装它，以内置的eslint:recommended推荐规则和NPM插件eslint-plugin-react的使用为例，我们在安装该插件后，需要添加这样的配置：

  • 引入插件需要通过plugins字段，可以省略包名的前缀 eslint-plugin-
  • 使用插件需要通过extends字段，它的值可以是数组，数组内的插件配置会进行合并，处在后面的插件规则的优先级更高
  • "extends": "plugin:react/recommended"使用的是eslint-plugin-react插件中的推荐规则集合，该插件还含有plugin:react/all规则集合
  • "extends": "plugin:react/recommended"继承的插件配置中其实已经有声明"plugins": "react"，所以这里配置可以省略plugins字段——"plugins": ["react"]

项目的代码检查机制

为了及早的发现错误的代码，防止错误代码被提交到Git，这里使用的是pre-commit时机的钩子来进行代码检查。

  1. 通过Husky挂载pre-commit钩子脚本"pre-commit": "node ./build/git/pre-commit.js",
  2. 在pre-commit.js脚本中执行Git命令git diff --cached --name-only --diff-filter=ACM -- '*.vue' '*.js' '*.jsx'，获取本次提交代码中的目标文件
  3. 通过文件的每行平均字符数，判断文件是否为打包生成的压缩文件
  4. 对非压缩的源码文件执行ESLint检查命令./node_modules/.bin/eslint --config .eslintrc.js --no-eslintrc ${filePath}
  5. 打印检查结果并判断脚本是否需要以非0值退出

常用Git命令

此处再列举一些本项目中使用到的一些比较有用的Git命令

  • git rev-parse HEAD 获取当前Git分支的hash
  • git rev-parse --revs-only HEAD 获取当前Git分支的名称 --revs-only也可以换成--abbrev-ref
  • git diff hashA hashB --name-only 获取两个hash值对应的提交的差异文件名，--name-only表示只返回文件名
  • git log -10 --graph --pretty=oneline --abbrev-commit 获取当前分支最近10次提交的记录美化后通过图表方式展示，-10表示数量限制为10条；--graph表示图表展示方式，否则为列表方式展示；--pretty=oneline表示去掉一些作者、日期等信息，把记录缩减为一行；--abbrev-commit表示显示部分hash
  • git config --global alias.lg "log --color --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit" 配置Git操作的别名，运行这条命令后，可以通过git lg来快速运行这条命令## 工程介绍

NDP平台采用的实例模板为static_append，即对已存在的同名资源覆盖、非同名资源增加的方式，不需要重启服务，所以效率和稳定性都比较高。

得益于本地构建、资源覆盖和差量部署几个特点，蜗牛读书的前端静态资源工程的一次部署只需要几秒钟。

差量部署原理：在NDP构建机上，利用构建脚本，查询一个在.gitignore列表中的配置文件（意味着该文件不会被上传和覆盖，只存在与构建机）。如果文件存在，则读取该文件中保存的上一次部署时工程的Git Hash，再结合本次部署的Git Hash，利用git diff命令，获得到两次hash间的文件差异从而进行最小化的差量部署；如果文件不存在或者两次hash相同，则进行全量部署，并且生成或更新配置文件。

附加功能

前端工程分离出来以后，就更便于添加一些其他的功能，比如分支规范检查、代码规范检查等

Git Hooks

Git Hooks是Git作为一个版本控制系统提供的在某些特定时机可以被触发的钩子，我可以在这些时机执行一些自定义脚本，这些自定义脚本可以做一些校验的操作，如果脚本以非零值退出，那么钩子将阻止本次操作。
Git Hooks有两类：客户端的和服务端的，客户端的钩子并不随代码提交，只在本地运行，而服务端的钩子只在远端工作。

常用客户端钩子

  • pre-commit 钩子在键入提交信息前运行。
  • prepare-commit-msg 钩子在启动提交信息编辑器之前，默认信息被创建之后运行。
  • commit-msg 在用户提交信息后触发，并且可以读取到提交的信息。
  • post-commit 钩子在整个提交过程完成后运行。

    常用服务端钩子

  • pre-receive 服务端收到来自客户端的推送操作时触发。
  • update 和 pre-receive 十分类似，不同之处在于如果一次推送到多个分支，update钩子会在每个分支都执行一次，而 pre-receive 只执行一次。
  • post-receive 在整个推送完结以后运行，可以用来更新其他系统服务或者通知用户。

提交信息的检查

有了Git Hooks之后，我们就可以对开发人员提交的Git Commit进行格式的检查，拒绝不符合格式的提交信息。
这里我们使用的就是commit-msg钩子。

Husky连接的钩子可以接收2个参数：

  • HUSKY_GIT_PARAMS代表该Git钩子运行时传入的参数，比如这里commit-msg钩子中就会有本次提交的message文件存放位置；
  • HUSKY_GIT_STDIN代表该Git钩子运行时从标准输入stdin中可以读取的一些信息，比如pre-push钩子中就会有一系列待更新的Git引用(git ref)

拿到commit message文件地址后，我们就可以读取本次提交的message内容了，配合以下我们团队制定的commit message规则进行校验即可：

  • feat：新功能（feature）
  • fix：修补bug
  • docs：文档（documentation）
  • style： 格式（不影响代码运行的变动）
  • refactor：重构（即不是新增功能，也不是修改bug的代码变动）
  • test：增加测试
  • chore：构建过程或辅助工具的变动

提交代码的检查

ESLint

我们使用ESLint可以配合Git Hooks进行代码检查，这里简要介绍一下Eslint：

重要概念
  • env 脚本的运行环境。每种环境都有一组特定的预定义全局变量,比如browser环境下的window
  • globals 脚本在执行期间访问的额外的全局变量，比如jquery的$
  • rules 启用的规则及其各自的错误级别，比如"no-console": 0/1/2/"off"/"warn"/"error"
  • plugins ESLint的插件，它可以是以上3类配置的任意组合，通过插件内的configs字段进行定义，而且configs字段还可以包含多个配置级别，比如all，代表各级别通用，recommended代表推荐级别，比all要多出一些规则和配置

rules在定义时，是通过分析Javascript代码转换成的AST树来实现的，这使得规则不但可以判断错误，还可以定义如何修复错误，一旦我们使用的ESLint命令行包含了–fix选项，ESLint就可以根据规则内置的错误修复逻辑直接帮我们修复错误。

配置文件

ESLint 支持几种格式的配置文件：

  • JavaScript - 使用 .eslintrc.js 然后输出一个配置对象。
  • YAML - 使用 .eslintrc.yaml 或 .eslintrc.yml 去定义配置的结构。
  • JSON - 使用 .eslintrc.json 去定义配置的结构，ESLint 的 JSON 文件允许 JavaScript 风格的注释。
  • (弃用) - 使用 .eslintrc，可以使 JSON 也可以是 YAML。
  • package.json - 在 package.json 里创建一个 eslintConfig属性，在那里定义你的配置。
    如果同一个目录下有多个配置文件，ESLint 只会使用一个。优先级顺序如下：
  1. .eslintrc.js
  2. .eslintrc.yaml
  3. .eslintrc.yml
  4. .eslintrc.json
  5. .eslintrc
  6. package.json

使用插件

ESLint支持的检查规则有非常的多，一个个去配置显然并不现实，所以我们需要使用插件。
ESLint的插件通常都是NPM包，我们需要先安装它，以内置的eslint:recommended推荐规则和NPM插件eslint-plugin-react的使用为例，我们在安装该插件后，需要添加这样的配置：

  • 引入插件需要通过plugins字段，可以省略包名的前缀 eslint-plugin-
  • 使用插件需要通过extends字段，它的值可以是数组，数组内的插件配置会进行合并，处在后面的插件规则的优先级更高
  • "extends": "plugin:react/recommended"使用的是eslint-plugin-react插件中的推荐规则集合，该插件还含有plugin:react/all规则集合
  • "extends": "plugin:react/recommended"继承的插件配置中其实已经有声明"plugins": "react"，所以这里配置可以省略plugins字段——"plugins": ["react"]
