Chrome浏览器中Cache-control：no-cache和no-store的区别

By Icewind

发表于2021-06-22

文章目录
  1. 1. 常规解释
  2. 2. 问题表现
  3. 3. 问题原因
  4. 4. 解决方式

HTTP Response Headers中的Cache-Control: no-cache和Cache-Control: no-store，看起来都是告知浏览器不要缓存该响应，但是他们是有区别的。有的时候如果没有使用合适的cache值，可能会达不到我们预想的目标。

常规解释

先来看下这2个值的常规区别解释：

  • no-cache: Somewhat of a misnomer, specifies that content can be cached but, if so, it must be re-validated on each request before being served to a client. This forces the client to check for freshness but allows it to avoid downloading the resource again if it has not changed. Mutually exclusive with no-store.
  • no-store: Indicates that the content actually cannot be cached in any way by any primary or intermediate cache. This is a good option for resources that may contain sensitive data, or for resources that will almost certainly change from visit to visit. Mutually exclusive with no-cache.

概括来说，就是这是2个互斥的Response Header：

  • no-cache并不是说这个资源完全不能被缓存，而是这个资源每次使用缓存前必须到服务端去验证一遍，如果是服务端返回304，代表资源是新鲜的，那么可以直接使用缓存。
  • no-store则更严格，它表示我们的浏览器在任何时候都不能缓存这个响应，必须重新请求。

有一点需要注意，如果设置了这2个值中的任意一个，源站到客户端的中间缓存节点都是不会缓存这个响应的。

问题表现

但是下面这个情况，在Chrome浏览器中no-cache的表现可能和我们想的不一样：

  • 假设我们当前访问是A页面，是一篇博客，博客上有一个关注按钮，这是一个SSR的页面，它的Response Header是中包含了Cache-Control: no-cache
  • 我们点击了A页面的这个关注按钮，发起异步请求，状态更新，这时关注按钮变成了已关注
  • 我们再点击A页面中的超链接，跳转到了B页面
  • 然后再点击浏览器的返回按钮，返回到了A页面
  • 此时我们可以看到，A页面的那个按钮状态显示的还是关注，而不是我们操作过的已关注
  • 我们刷新A页面，按钮的状态恢复正常，显示已关注

这就是Chrome浏览器很早就遗留下的表现，与我们通常认为的no-cache机制不太一样。但是由于也没有关于浏览器前进、后退的缓存机制的严格规定，而且大部分情况下，这样的机制是可以给用户更快的响应展现的，所以也不能说是一个问题。

  • Response Headers中含有Cache-Control: no-cache

  • 在浏览器返回时直接使用了缓存，并没有验证其新鲜度

问题原因

  • 浏览器返回时，浏览器直接提供的是之前缓存的页面
    • 我们可以从浏览器devtool-network下看到，在返回时，浏览器直接使用的是disk cache，而没有经过任何新鲜度的检查，这与no-cache的定义好像不符
  • 这不是bfcache
    • bfcache是浏览器在返回时直接使用内存中的页面，这种情况下，被缓存的页面不会触发onLoad和DOMContentLoaded事件的，只会触发pageShow等事件，而且可以在这个事件中拿到标识event.persisted，来判断页面呈现是否由bfcache触发。
    • 上面的场景中，页面的onLoad、DOMContentLoaded、pageShow等事件是正常触发的，而event.persisted标识一直为false

解决方式

面对上面的那个问题，不设置Response Headers中的Cache-Control、设置Cache-Control: no-cache亦或者是Cache-Control: max-age=0都不能解决。

此时就需要使用Cache-Control: no-store的Response Headers了。这样设置，浏览器才不会储存这个SSR页面，从而在返回时也去拉取最新的结果。

注意事项

一旦设置了Cache-Control: no-store，浏览器就真的完全不存这个响应了，这意味着你右键点击-显示网页源代码，浏览器也都没有可缓存的源代码供你查看，而是会重新发起这个网页的请求才能拿到源代码来显示。

所以，如果不是一些必要的场景，比如上面的bug fix，no-store还是要慎用的。
